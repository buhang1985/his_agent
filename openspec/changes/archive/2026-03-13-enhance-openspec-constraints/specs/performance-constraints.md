## 新增需求

### 需求：并发请求限制

系统必须实施多层并发限制，保护系统稳定性。

#### 场景：全局并发限制
- **当** 系统处理请求时
- **那么** 必须遵守：
  - 最大并发请求数：100
  - 最大等待队列：50
  - 超出限制返回 HTTP 429 Too Many Requests
  - 等待队列满时返回 HTTP 503 Service Unavailable

#### 场景：单用户并发限制
- **当** 单个用户发起请求时
- **那么** 必须限制：
  - 每秒最大请求数：10
  - 最大并发会话数：3
  - 每用户等待队列：5
  - 超出限制返回 HTTP 429

#### 场景：接口级并发限制
- **当** 访问特定接口时
- **那么** 必须限制：
  - 语音转写接口：每秒 5 次（受外部 API 限制）
  - 病历生成接口：每秒 2 次（LLM 资源密集）
  - HIS 同步接口：每秒 10 次

### 需求：超时控制

所有外部调用和内部处理必须设置超时，防止资源占用。

#### 场景：外部服务超时
- **当** 调用外部服务时
- **那么** 必须设置：
```yaml
timeouts:
  speech-recognition: 5s     # 语音识别
  llm-generation: 30s        # LLM 病历生成
  his-integration: 10s       # HIS 系统调用
  database-query: 3s         # 数据库查询
  redis-operation: 1s        # 缓存操作
```

#### 场景：内部处理超时
- **当** 执行内部业务逻辑时
- **那么** 必须设置：
  - 请求总处理时间：≤ 60s
  - 单个业务步骤：≤ 10s
  - 事务执行时间：≤ 5s
  - 超出超时自动回滚并记录日志

#### 场景：超时处理
- **当** 发生超时时
- **那么** 必须：
  - 释放已占用资源
  - 记录超时日志（包含 traceId）
  - 返回友好错误提示
  - 触发降级策略（如适用）

### 需求：资源配额管理

系统资源必须按优先级分配，保证关键业务可用。

#### 场景：线程池配额
- **当** 配置线程池时
- **那么** 必须：
```yaml
thread-pools:
  # 核心业务线程池
  core-business:
    core-size: 20
    max-size: 50
    queue-capacity: 100
    
  # 异步任务线程池
  async-tasks:
    core-size: 5
    max-size: 10
    queue-capacity: 50
    
  # 定时任务线程池
  scheduled-tasks:
    core-size: 5
    max-size: 5
    queue-capacity: 0
```

#### 场景：内存配额
- **当** 分配内存资源时
- **那么** 必须：
  - JVM 堆内存：4GB（生产环境）
  - 单个请求内存：≤ 50MB
  - 缓存占用：≤ 1GB
  - 录音缓存：≤ 100MB

#### 场景：数据库连接池
- **当** 配置数据库连接时
- **那么** 必须：
```yaml
spring.datasource.hikari:
  maximum-pool-size: 20
  minimum-idle: 5
  connection-timeout: 30000
  idle-timeout: 600000
  max-lifetime: 1800000
```

### 需求：性能指标要求

系统性能必须达到以下指标要求。

#### 场景：响应时间要求
- **当** 处理请求时
- **那么** 必须满足：
  - P50 响应时间：< 200ms
  - P95 响应时间：< 500ms
  - P99 响应时间：< 1000ms
  - 病历生成总时间：< 10s

#### 场景：吞吐量要求
- **当** 系统运行时
- **那么** 必须支持：
  - 并发用户数：≥ 100
  - 每秒请求数：≥ 50
  - 语音转写并发：≥ 10 路
  - 日均问诊量：≥ 1000 次

#### 场景：可用性要求
- **当** 系统运行时
- **那么** 必须保证：
  - 系统可用性：≥ 99.5%
  - 语音识别成功率：≥ 95%
  - 病历生成成功率：≥ 98%
  - 数据持久化成功率：100%
