# 缓存策略规范

**版本**: 1.0  
**日期**: 2026-03-11  
**状态**: 新增

---

## 新增需求

### 需求：统一缓存存储

系统必须使用 Redis 作为统一的缓存存储解决方案，禁止使用多种缓存存储混用。

#### 场景：缓存选型
- **当** 需要实现缓存功能时
- **那么** 必须使用 Redis 7+ 作为缓存存储
- **那么** 禁止使用 HashMap、ConcurrentHashMap 等本地缓存作为主要缓存

#### 场景：缓存依赖
- **当** 配置项目依赖时
- **那么** 必须添加 Spring Data Redis 依赖
- **那么** 必须配置 Redis 连接信息（host、port、password、database）

---

### 需求：缓存键命名规范

所有缓存键必须遵循统一的命名规范，便于管理和排查问题。

#### 场景：缓存键格式
- **当** 创建缓存键时
- **那么** 必须遵循格式：`his_agent:{模块}:{数据类型}:{ID}`
- **那么** 模块名使用小写字母和下划线（如：`user`、`consultation`）
- **那么** 数据类型使用小写字母（如：`info`、`list`、`config`）

#### 场景：缓存键示例
- **当** 缓存患者信息时
- **那么** 键名为：`his_agent:patient:info:{patientId}`
- **当** 缓存用户会话时
- **那么** 键名为：`his_agent:user:session:{userId}`
- **当** 缓存医学词库时
- **那么** 键名为：`his_agent:medical:term:{termType}`

---

### 需求：缓存过期时间管理

所有缓存必须设置明确的过期时间，禁止永久缓存。

#### 场景：过期时间设置
- **当** 写入缓存时
- **那么** 必须设置过期时间（TTL）
- **那么** 禁止使用 -1 或 0 表示永不过期

#### 场景：数据类型过期时间
- **当** 缓存患者数据时
- **那么** 过期时间必须 ≤ 30 分钟
- **当** 缓存医学词库时
- **那么** 过期时间必须 ≤ 24 小时
- **当** 缓存用户会话时
- **那么** 过期时间必须 ≤ 2 小时
- **当** 缓存配置数据时
- **那么** 过期时间必须 ≤ 1 小时
- **当** 缓存 API 响应时
- **那么** 过期时间必须 ≤ 10 分钟

#### 场景：过期时间随机偏移
- **当** 设置批量缓存过期时间时
- **那么** 必须添加随机偏移（±20%）
- **那么** 防止缓存雪崩（同一时间大量缓存失效）

---

### 需求：缓存穿透防护

系统必须实现缓存穿透防护措施，避免查询不存在的数据。

#### 场景：布隆过滤器
- **当** 查询量大且数据量庞大时
- **那么** 必须使用布隆过滤器预判数据是否存在
- **那么** 布隆过滤器判断不存在时直接返回，不查询数据库

#### 场景：空值缓存
- **当** 查询数据不存在时
- **那么** 必须缓存空值（null 或特殊标记）
- **那么** 空值缓存过期时间 ≤ 5 分钟
- **那么** 防止恶意查询不存在的数据

---

### 需求：缓存雪崩防护

系统必须实现缓存雪崩防护措施，避免大量缓存同时失效。

#### 场景：过期时间随机化
- **当** 设置缓存过期时间时
- **那么** 必须在基础时间上添加随机偏移
- **那么** 随机偏移范围：±20%
- **例如** 基础 30 分钟，实际过期时间为 24-36 分钟随机

#### 场景：热点数据永不过期（逻辑上）
- **当** 数据为热点数据（如医学词库）时
- **那么** 必须实现后台定时刷新
- **那么** 在过期前主动更新缓存
- **那么** 避免热点数据失效导致数据库压力

---

### 需求：缓存更新策略

系统必须实现一致的缓存更新策略，保证数据一致性。

#### 场景：Cache-Aside 模式
- **当** 读取数据时
- **那么** 先读缓存，命中则返回
- **那么** 未命中则读数据库，写入缓存后返回

#### 场景：数据更新
- **当** 更新数据时
- **那么** 必须先更新数据库
- **那么** 再删除缓存（不是更新缓存）
- **那么** 禁止先更新缓存再更新数据库

#### 场景：缓存删除失败
- **当** 删除缓存失败时
- **那么** 必须记录错误日志
- **那么** 必须重试最多 3 次
- **那么** 重试失败后发送告警

---

### 需求：本地缓存（可选）

系统可使用 Caffeine 作为本地缓存，用于缓存极少变化的数据。

#### 场景：本地缓存使用
- **当** 数据极少变化且访问频繁时
- **那么** 可使用 Caffeine 本地缓存
- **例如** 医院科室列表、字典数据
- **那么** 本地缓存过期时间 ≤ 1 小时

#### 场景：本地缓存 + Redis 双层缓存
- **当** 使用双层缓存时
- **那么** 先查本地缓存
- **那么** 未命中则查 Redis 缓存
- **那么** 仍未命中则查数据库并写入两层缓存

---

### 需求：缓存监控

系统必须实现缓存监控，便于观察缓存运行状态。

#### 场景：缓存指标
- **当** 系统运行时
- **那么** 必须暴露缓存指标：
  - 缓存命中率
  - 缓存键数量
  - 缓存内存使用量
  - 缓存删除/更新次数

#### 场景：缓存告警
- **当** 缓存命中率 < 50% 时
- **那么** 记录警告日志
- **当** 缓存内存使用 > 80% 时
- **那么** 发送告警通知
- **当** Redis 连接失败时
- **那么** 立即发送 P0 告警

---

### 需求：技术选型明确

系统必须明确前后端技术栈，统一开发框架。

#### 场景：前端技术栈
- **当** 开发前端功能时
- **那么** 必须使用：
  - Vue 3.4+（ Composition API）
  - TypeScript 5+（严格模式）
  - Vite 6+（构建工具）
  - Element Plus 2.8+（UI 组件库）
  - Pinia 2.2+（状态管理）

#### 场景：后端技术栈
- **当** 开发后端功能时
- **那么** 必须使用：
  - Java 17+（LTS 版本）
  - Spring Boot 3.x
  - Maven 3.9+（构建工具）
  - Spring Data Redis（Redis 客户端）
  - Lombok（代码简化）

#### 场景：数据库选型
- **当** 使用关系型数据库时
- **那么** 必须使用 MySQL 8+
- **那么** 禁止使用其他关系型数据库（PostgreSQL、Oracle 等）
- **那么** 字符集必须使用 UTF8MB4

## 验收标准

- [ ] 所有缓存操作使用 Redis
- [ ] 缓存键命名符合规范
- [ ] 所有缓存设置过期时间
- [ ] 实现缓存穿透防护（布隆过滤器或空值缓存）
- [ ] 实现缓存雪崩防护（随机过期时间）
- [ ] 缓存更新遵循 Cache-Aside 模式
- [ ] 缓存监控指标可观察
- [ ] 技术栈符合选型规范（Vue + Java + MySQL）
