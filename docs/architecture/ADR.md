# 架构决策记录

## ADR-001: 本地化部署 + 云端 AI 服务

**日期**: 2026-03-11  
**状态**: 已批准  
**决策者**: 项目团队

### 背景

医疗行业对数据主权和合规性有严格要求，需要平衡：
- 患者数据留存院内（等保三级要求）
- 利用先进 AI 能力（语音识别、LLM）
- 系统高可用性和易维护性

### 决策

采用**本地物理服务器部署 + 云端 AI 服务**的混合架构：

**本地部署**:
- his_agent 应用（Spring Boot）
- MySQL 数据库
- Redis 缓存
- Nginx 反向代理

**云端服务**:
- 讯飞医疗 ASR（语音识别）
- 阿里云 Qwen（LLM 病历生成）
- 腾讯云 ASR（备选降级）

### 优势

✅ 患者数据不出院，满足等保要求  
✅ 利用云端 AI 最佳能力  
✅ 降低初期建设成本（无需 GPU 服务器）  
✅ 弹性扩展 AI 能力

### 劣势

⚠️ 依赖医院外网出口稳定性  
⚠️ 云端 API 调用成本  
⚠️ 需要防火墙出站策略

### 缓解措施

- 部署 Keepalived 高可用，故障切换 < 3 秒
- 配置熔断降级，云端 API 失败自动切换备选方案
- 防火墙仅开放出站 HTTPS 443

## ADR-002: Keepalived 抢占模式高可用

**日期**: 2026-03-11  
**状态**: 已批准  
**决策者**: 项目团队

### 背景

需要双机热备方案，在 Keepalived 抢占模式和非抢占模式之间选择。

### 决策

采用**抢占模式**：

- Node 1 配置为 MASTER（priority 100）
- Node 2 配置为 BACKUP（priority 90）
- Node 1 恢复后，VIP 自动切回

### 理由

1. **资源利用最优化**: 主节点性能通常更强，应优先承载流量
2. **运维可预测性**: 正常状态下流量固定到主节点
3. **故障恢复明确**: 主节点恢复后自动接管，无需人工干预

### 风险

- 频繁故障可能导致 VIP 反复漂移

### 缓解措施

- 配置 `fall 3` 和 `rise 1` 避免抖动
- 设置合理的 `advert_int`（1 秒）

## ADR-003: HIS Token 公钥验证方案

**日期**: 2026-03-11  
**状态**: 已批准  
**决策者**: 项目团队

### 背景

HIS 系统分配 Token 给 his_agent，需要验证 Token 真实性。三种方案：
- A: 公钥验证（非对称加密）
- B: 共享密钥（HMAC）
- C: 回调验证（实时查询 HIS）

### 决策

采用**方案 A: 公钥验证**

### 理由

1. **解耦最好**: his_agent 无需访问 HIS 即可验证
2. **性能最优**: 本地验证，无网络延迟
3. **安全性高**: 私钥仅在 HIS，公钥泄露不影响安全
4. **可用性高**: HIS 故障不影响 his_agent 认证

### 技术实现

- HIS 使用 RSA-2048 私钥签名 JWT
- his_agent 使用公钥验证 JWT
- Token 过期时间：1 小时
- Token 刷新：postMessage 通知 HIS

## ADR-004: 可配置备份策略

**日期**: 2026-03-11  
**状态**: 已批准  
**决策者**: 项目团队

### 背景

不同医院对备份策略要求不同：
- 部分医院要求本地备份即可
- 部分医院要求异地备份
- 保留时长要求不同

### 决策

采用**可配置备份策略**，通过配置文件控制：

```yaml
backup:
  mode: both  # local | remote | both
  local:
    enabled: true
    retention-days: 30
  remote:
    enabled: true
    type: rsync  # rsync | sftp | s3
    host: backup.hospital.com
```

### 优势

✅ 灵活性高，适配不同医院要求  
✅ 配置简单，无需修改代码  
✅ 支持渐进式部署（测试期本地，生产期异地）

## ADR-005: 独立监控部署

**日期**: 2026-03-11  
**状态**: 已批准  
**决策者**: 项目团队

### 背景

监控系统部署方案选择：
- 方案 A: 与 his_agent 同机部署
- 方案 B: 独立监控集群

### 决策

采用**方案 B: 独立监控部署**

### 理由

1. **故障隔离**: his_agent 故障不影响监控
2. **资源隔离**: 监控不占用应用资源
3. **可扩展性**: 可监控多个 his_agent 实例
4. **运维独立**: 监控团队独立管理

### 技术选型

- Prometheus: 指标收集
- Grafana: 可视化
- Alertmanager: 告警管理
- Node Exporter: 服务器指标
- MySQL Exporter: 数据库指标
- Redis Exporter: 缓存指标

## ADR-006: 测试期单机 MySQL，生产期主从复制

**日期**: 2026-03-11  
**状态**: 已批准  
**决策者**: 项目团队

### 背景

平衡测试期成本和生产期高可用需求。

### 决策

**测试期**: 单机 MySQL
- 快速部署，降低成本
- 每日备份 + binlog 增量备份

**生产期**: MySQL 主从复制
- 1 主 1 从，读写分离
- 自动故障切换（手动提升从库）

### 迁移路径

1. 测试期：单机 MySQL，配置 binlog
2. 生产期：部署从库，配置复制
3. 应用层：配置读写分离（可选）

## ADR-007: 容器化封装

**日期**: 2026-03-11  
**状态**: 已批准  
**决策者**: 项目团队

### 决策

采用 Docker 容器化封装：

- 应用：Docker 镜像
- 部署：Docker Compose
- 配置：环境变量外部化

### 理由

✅ 环境一致性（开发/测试/生产）  
✅ 快速部署和回滚  
✅ 资源隔离和限制  
✅ 便于迁移和扩容

### 镜像规范

- 基础镜像：eclipse-temurin:17-jdk-alpine
- 非 root 用户运行
- 健康检查配置
- JVM 参数优化
